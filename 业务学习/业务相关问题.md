## 钢铁行业质量管理体系关联分析

### 判定标准与检验标准：质量目标与执行依据

#### 关联性

- **判定标准** 是产品质量的最终要求，定义了产品是否合格的边界条件
- **检验标准** 是实现判断标准的操作指南，明确检验方法、抽样规则、检测设备和判定流程

#### 协同作用

- 检验标准需严格对标判断标准，确保检测结果能准确反映产品是否符合质量目标。
- 若判定标准调整，检验标准需同步更新检测方法和验收阈值

### 组批管理与检验委托：过程管控与实施操作

- **组批管理** 是对生产批次进行逻辑分组（如按炉次、轧制批次或订单号），赋予唯一标识，确保产品可追溯。
- **检验委托** 是基于组批信息触发检验任务，明确需要检测的批次、检验项目及执行部门

- 组批管理为检验委托提供基础数据（如批次号、生产时间、工艺参数），确保检验对象清晰可溯。
- 组批规则（如每10吨钢材为一个检验批次）直接影响检验委托的频次和样本量。
- 若组批异常（如某炉次成分波动），检验委托需动态调整

### 检验标准与检验委托：执行依据到具体任务

- 检验标准规定"如何检验",检验委托则落实"何时检、检什么、谁负责"。
- 例如：检验标准规定"每批次取3个试样进行拉伸试验",检验委托则生成具体任务单，指派实验室执行。

- 检验委托的生成必须严格遵循检验标准的抽样规则和检测要求，避免漏检或误检。
- 若检验标准要求多项目检测（如成分+硬度+表面缺陷），检验委托需同步覆盖所有项目。

### 质量实绩与各环节：结果反馈与持续改进

- **质量实绩**是检验结果的汇总（如合格率、缺陷类型分布、性能数据），反映实际质量水平。
- 质量实绩数据需与判定标准对比（是否达标）、与组批管理关联（问题批次定位）、与检验标准复盘（检测方法有效性）。
- 质量实绩用于验证判定标准的合理性（如客户投诉是否因标准过低）和检验标准的有效性（如漏检率是否可控）
- 通过分析质量实绩与组批的关联，可追溯生产环节问题
- 质量实绩为优化组批规则（如调整批次大小）和检验委托策略（如高风险批次加严检验）提供数据支持。

## 组批形式只有汇总量（结算量）而没有毛重、皮重、净重的理解

### 📊 **组批管理的核心概念**

组批管理是一种**汇总处理模式**，它将多个单车作业按照批次号进行汇总，然后进行统一的财务抛帐处理。

### 🔄 **组批 vs 单车的区别**

#### **单车管理（MAT_BATCH_FLAG = "N"）**

```cpp
// 单车抛账：查询单条作业记录
comm_inq.SetCommandText(" SELECT * FROM TYMREC1 WHERE WORK_SEQ_NO=@WORK_SEQ_NO ");
```

- **数据来源**：单条作业记录
- **重量信息**：完整的毛重、皮重、净重、结算量
- **处理方式**：逐条处理，每条记录都有完整的重量信息

#### **组批管理（MAT_BATCH_FLAG = "Y"）**

```cpp
// 组批抛账：汇总查询
CString pc_sql = " SELECT SUM(BALANCE_WGT) BALANCE_WGT,BUSY_TYPE,DST_STOCK_CODE,"
                 " DST_MAT_CODE,BUY_ORDER_NO,TASKS_NO,IN_BATCH_NO  "
                 " FROM TYMREC1 WHERE IN_BATCH_NO=@WORK_SEQ_NO "
                 " AND BUSY_TYPE LIKE '%'||'采购'||'%'  "
                 " GROUP BY BUSY_TYPE,DST_STOCK_CODE,DST_MAT_CODE,BUY_ORDER_NO,TASKS_NO,IN_BATCH_NO  ";
```

- **数据来源**：多条作业记录的汇总
- **重量信息**：只有汇总后的结算量
- **处理方式**：按批次汇总处理

### 🎯 **为什么组批只有结算量？**

#### **1. 业务逻辑层面**

**组批管理的本质**：

- 将多个单车作业（可能来自不同的磅单）按批次号进行逻辑分组
- 每个批次可能包含多个车次的进厂作业
- 财务核算时只需要知道批次的总结算量

**举例说明**：

```C
批次号：B20241201001
├── 车次1：毛重1000kg，皮重200kg，净重800kg，结算量750kg
├── 车次2：毛重1200kg，皮重250kg，净重950kg，结算量900kg
├── 车次3：毛重1100kg，皮重220kg，净重880kg，结算量830kg
└── 批次汇总：结算量 = 750 + 900 + 830 = 2480kg
```

#### **2. 技术实现层面**

**SQL汇总查询**：

```sql
SELECT SUM(BALANCE_WGT) BALANCE_WGT,  -- 只汇总结算量
       BUSY_TYPE,DST_STOCK_CODE,DST_MAT_CODE,BUY_ORDER_NO,TASKS_NO,IN_BATCH_NO
FROM TYMREC1 
WHERE IN_BATCH_NO=@WORK_SEQ_NO 
  AND BUSY_TYPE LIKE '%采购%'
GROUP BY BUSY_TYPE,DST_STOCK_CODE,DST_MAT_CODE,BUY_ORDER_NO,TASKS_NO,IN_BATCH_NO
```

**关键点**：

- `SUM(BALANCE_WGT)`：只对结算量进行汇总
- 没有对毛重、皮重、净重进行汇总
- 因为财务核算主要关注结算量

#### **3. 重量字段赋值逻辑**

**组批管理下的赋值**：

```cpp
// 第347-353行：重量字段赋值
TYMREE1["GROSS_WT"] = TYMREC1["GROSS_WT"];       // 来自汇总查询结果
TYMREE1["TARE_WT"] = TYMREC1["TARE_WT"];         // 来自汇总查询结果
TYMREE1["NET_WT"] = TYMREC1["NET_WT"];           // 来自汇总查询结果
TYMREE1["BALANCE_WGT"] = TYMREC1["BALANCE_WGT"]; // 来自汇总查询结果
```

**问题**：汇总查询中只有 `SUM(BALANCE_WGT)`，没有其他重量字段的汇总，所以：

- `GROSS_WT`、`TARE_WT`、`NET_WT` 字段在汇总查询中为空或无效值
- 只有 `BALANCE_WGT` 字段有有效的汇总值

### 💡 **业务意义理解**

#### **1. 财务核算需求**

- **财务关注点**：批次的总结算量（用于成本核算）
- **计量关注点**：单车作业的详细重量信息（用于计量管理）
- **组批抛帐**：主要服务于财务核算，因此只需要结算量

#### **2. 数据完整性**

- **单车层面**：保留完整的重量信息，用于计量管理和追溯
- **批次层面**：只保留财务核算必需的结算量信息
- **数据冗余**：避免在批次层面重复存储单车已存在的重量信息

#### **3. 业务追溯**

- **批次抛帐记录**：记录批次的总结算量和财务信息
- **单车作业记录**：保留详细的重量信息和计量数据
- **追溯关系**：通过批次号可以追溯到具体的单车作业

### 🔧 **实际应用场景**

#### **采购进厂业务**

1. **单车作业**：每车进厂都有完整的磅单数据（毛重、皮重、净重、结算量）
2. **批次组批**：将同一采购订单的多个车次按批次号分组
3. **批次抛帐**：按批次汇总结算量进行财务抛帐
4. **数据存储**：
   - `TYMREC1`：存储单车详细重量信息
   - `TYMREE1`：存储批次汇总的结算量信息

#### **质量检验业务**

1. **批次检验**：按批次进行质量检验
2. **检验结果**：生成质检重量（CHECK_WGT_1、CHECK_WGT_2）
3. **质检抛帐**：使用质检重量作为结算量进行抛帐
4. **重量信息**：质检业务不涉及磅单计量，因此没有毛重、皮重、净重

### �� **总结**

组批形式只有汇总量（结算量）而没有毛重、皮重、净重的原因是：

1. **业务逻辑**：组批管理是汇总处理模式，主要服务于财务核算
2. **技术实现**：SQL汇总查询只对结算量进行汇总，不汇总其他重量字段
3. **数据设计**：避免数据冗余，单车层面保留详细重量信息，批次层面只保留财务必需信息
4. **追溯关系**：通过批次号可以追溯到具体的单车作业，获取完整的重量信息

这种设计既满足了财务核算的需求，又保持了数据的完整性和可追溯性。
