# 副原料批次收货撤回业务流程详解

## 一、业务场景与前提条件

### 业务场景

已收料确认并已抛帐/暂估的"副原料"批次，需要撤回整个批次的收货操作。

### 资源类型

`BASE_RESOURCE = '副原料'`

### 涉及的数据库表

- **TYMREPCINFO**：批次信息表（批次头）
- **TYMREC1**：车运作业明细表（批次下所有车辆）
- **TQMRE27**：检验委托表（关联批次检验）
- **TYMREE1**：抛帐履历表（记录抛帐/红冲记录）

---

## 二、详细操作流程（按代码执行顺序）

### 阶段1：前置校验

#### 1.1 时间窗口检查（代码第45-53行）

```cpp
if (WORK_DATE == c_last_day)  // 每月最后一天
{
    if (hh == "22" || hh == "23")  // 22点后
    {
        → 报错：每个月最后一天22点后，禁止进厂收货！
    }
}
```

**校验目的：** 保护月末结账，避免影响财务核算期间。

---

#### 1.2 批次号基础校验（代码第75-93行）

**校验项：**

- 批次号不能为空
- 批次必须在 `TYMREPCINFO` 表中存在

```cpp
if (IN_BATCH_NO == "")
    → 报错：批次为空，不能进行批次收货撤回

if (TYMREPCINFO.Query("IN_BATCH_NO") == false)
    → 报错：批次查询失败，不能进行批次收货撤回
```

---

#### 1.3 批次状态校验（代码第96-131行）

##### 委托状态检查

```cpp
if (ENTR_FLAG == "00")  // 编制中
    → 报错：编制中不能进行批次收货撤回
```

##### 采购结算状态检查

```cpp
if (PSCS_PC_STATUS == "Y")  // 已结算
    → 报错：采购已结算，不能进行批次收货撤回
```

##### 抛帐状态检查（关键）

**允许撤回的状态：**

- `THROW_FLAG = 'T'`（部分退货）
- `THROW_FLAG = 'Y'`（已抛帐）
- `THROW_FLAG = 'Z'`（暂估）

**不允许撤回的状态：**

```cpp
if (THROW_FLAG != "T" && THROW_FLAG != "Y" && THROW_FLAG != "Z")
    → 报错：已抛帐、部分退货、已抛帐、暂估才允许批次收货撤回

if (THROW_FLAG == "X")  // 不需要抛帐
    → 报错：不需要抛帐，不能进行批次收货撤回
```

---

#### 1.4 批次作业数据校验（代码第133-151行）

**校验项1：必须存在对应作业**

```cpp
if (TYMREC1.QueryCount("IN_BATCH_NO") == 0)
    → 报错：批次无对应的进厂作业，不能进行批次收货撤回
```

**校验项2：发料确认状态检查**

```cpp
// 必须所有车都已完成发料确认（不允许有未确认的）
if (存在 SEND_CONFM_FLAG='N' 的记录)
    → 报错：存在未发料确认的进厂作业，不能进行批次收货撤回
```

---

### 阶段2：核心撤回处理（代码第163-241行）

**前提条件：** `BUY_ORDER_NO` 不为空（采购业务）

#### 2.1 红冲处理（代码第172-186行）

**触发条件：** `THROW_FLAG` 为 `'Y'`、`'Z'` 或 `'T'` 时

```cpp
if (THROW_FLAG == "Z" || THROW_FLAG == "T" || THROW_FLAG == "Y")
{
    // 生成抛账履历（红冲）
    doFlag = f_ymre_toERP(IN_BATCH_NO, "1", "Y", conn);
    
    if (doFlag != 0)
        → 报错：抛账履历生成失败！
}
```

**作用：**

- 生成红字抛账履历（写入 `TYMREE1` 或相关抛帐履历表）
- 冲销原抛帐记录，确保财务账实相符
- **失败则整个事务回滚，返回错误**

---

#### 2.2 发送批次汇总抛帐撤回（代码第188-194行）

```cpp
// 通知采购系统：批次汇总抛帐撤回
doFlag = f_zy_cgjczp_snd(IN_BATCH_NO, "D", conn);
// DEAL_FLAG='D' 表示删除/撤回操作

if (doFlag != 0)
    → 报错并回滚
```

**作用：**

- 通知采购系统该批次汇总抛帐已撤回
- 确保外部系统数据同步

---

#### 2.3 更新批次状态（代码第198-219行）

##### 更新 TYMREPCINFO（批次信息表）

```cpp
if (THROW_FLAG == "Y" || THROW_FLAG == "Z")
{
    THROW_FLAG = "N";  // 回退为未抛帐状态
}
ZANGU_FLAG = "";  // 清空暂估标志

// 更新字段：ZANGU_FLAG, THROW_FLAG, REC_REVISOR, REC_REVISE_TIME
TYMREPCINFO.Update("ZANGU_FLAG,THROW_FLAG,REC_REVISOR,REC_REVISE_TIME", "IN_BATCH_NO");
```

##### 更新 TQMRE27（检验委托表）

```cpp
// 同步更新检验委托表的抛帐状态
TQMRE27["THROW_FLAG"] = "N";
TQMRE27["ZANGU_FLAG"] = "";
TQMRE27.Update("ZANGU_FLAG,THROW_FLAG,REC_REVISOR,REC_REVISE_TIME", "IN_BATCH_NO");
```

**字段变化说明：**

- `THROW_FLAG`: `'Y'/'Z'` → `'N'`（未抛帐）
- `ZANGU_FLAG`: 清空（暂估标志移除）
- `REC_REVISOR`: 更新为当前操作人
- `REC_REVISE_TIME`: 更新为当前时间戳

---

#### 2.4 发送单车收料实绩撤回（代码第221-236行）

**查询批次下所有车运作业：**

```sql
SELECT * FROM TYMREC1 
WHERE IN_BATCH_NO = @IN_BATCH_NO 
  AND BUY_ORDER_NO != '' 
  AND BUSY_TYPE IN ('采购进厂','采购直供','退货')
```

**对每辆车执行：**

```cpp
while (comm_inq.Read())
{
    comm_inq.Fetch(TYMREC1);
    
    // 通知采购系统：单车进厂收料实绩撤回
    doFlag = f_zy_cgdsj_snd(WORK_SEQ_NO, "D", conn);
    // DEAL_FLAG='D' 表示删除/撤回操作
    
    if (doFlag != 0)
        → 报错并回滚
}
```

**作用：**

- 逐车通知采购系统该车的收料实绩已撤回
- 保证单车数据与批次汇总数据一致
- **注意：** 代码注释中提到"合金单车"，实际适用于所有资源类型（包括副原料）

---

### 阶段3：副原料特殊处理（代码第243-260行）

**重要发现：代码中没有针对"副原料"的特殊逻辑！**

**代码中只有"废钢"的特殊处理：**

```cpp
// 仅当 BASE_RESOURCE == "废钢" 且品名不是 FGN21/FGN22 时
// 才会将 TYMREC1.RECV_CONFM_FLAG 置为 'N'
if (BASE_RESOURCE == "废钢" && MAT_CODE != "FGN21" && MAT_CODE != "FGN22")
{
    UPDATE TYMREC1 
    SET RECV_CONFM_FLAG='N', 
        REC_REVISOR=@USER, 
        REC_REVISE_TIME=@TIME 
    WHERE IN_BATCH_NO=@IN_BATCH_NO 
      AND BUSY_TYPE='采购进厂';
}
```

**对于副原料：**

- ✅ 不执行 `RECV_CONFM_FLAG` 的回退更新
- ✅ 收料确认标志保持原值（通常仍为 `'Y'`）
- ✅ 撤回主要体现在批次抛帐状态的回退和外部系统通知

## 六、业务含义说明

### 副原料批次收货撤回的含义

1. **批次层面：** 抛帐状态回退，财务账通过红冲处理
2. **单车层面：** 不强制回退收料确认标志（保留业务痕迹）
3. **外部系统：** 同步通知采购系统撤回

### 适用场景

- ✅ 批次抛帐后发现数据有误
- ✅ 需要调整批次组合
- ✅ 临时撤回待重新处理

### 注意事项

- ⚠️ 若批次已结算（`PSCS_PC_STATUS='Y'`），不允许撤回
- ⚠️ 若存在未发料确认的作业，不允许撤回
- ⚠️ 月末最后一天22点后禁止撤回（保护结账）

## 十一、红冲未生成的常见原因排查

### 11.1 红冲生成的核心条件

根据 `f_ymre_toERP` 函数代码分析，批次红冲需要满足以下条件：

```cpp
// 查询原抛帐记录（用于获取红冲量）
SELECT BALANCE_WGT, H2O_NET, H2O, ERP_SEQ_NO 
FROM TYMREE1 
WHERE IN_BATCH_NO = @IN_BATCH_NO 
  AND BALANCE_WGT > 0           // 必须大于0
  AND TRADE_TYPE = '01B'        // 交易类型必须为01B
  AND ENABLE_FLAG = '1'         // 必须是有效记录
  AND DST_STOCK_CODE = @DST_STOCK_CODE  // 目的库区必须匹配
ORDER BY REC_CREATE_TIME DESC 
FETCH FIRST 1 ROWS ONLY
```

### 11.2 可能原因分析

#### 原因1：未找到原抛帐记录（最常见）

**问题描述：** `TYMREE1` 表中找不到对应批次号的原抛帐记录

**可能情况：**

- ❌ 原批次抛帐时记录未正确插入 `TYMREE1` 表
- ❌ 原抛帐记录的 `TRADE_TYPE` 不是 `'01B'`
- ❌ 原抛帐记录的 `ENABLE_FLAG` 不是 `'1'`（已被禁用）
- ❌ 原抛帐记录的 `BALANCE_WGT` ≤ 0（异常数据）
- ❌ 原抛帐记录的 `DST_STOCK_CODE` 与撤回时查询的不匹配

**排查SQL：**

```sql
-- 检查批次是否有原抛帐记录
SELECT * 
FROM TYMREE1 
WHERE IN_BATCH_NO = '批次号' 
  AND BALANCE_WGT > 0 
  AND ENABLE_FLAG = '1'
ORDER BY REC_CREATE_TIME DESC;

-- 检查所有相关记录（包括禁用的）
SELECT IN_BATCH_NO, BALANCE_WGT, TRADE_TYPE, ENABLE_FLAG, 
       DST_STOCK_CODE, HC_FLAG, REC_CREATE_TIME
FROM TYMREE1 
WHERE IN_BATCH_NO = '批次号'
ORDER BY REC_CREATE_TIME DESC;
```

#### 原因2：订单号为空，跳过了红冲分支

**问题描述：** `BUY_ORDER_NO` 为空，代码不会进入红冲处理分支

**代码位置：** `qmre22pc_rec.cpp` 第165行

```cpp
if (TYMREPCINFO["BUY_ORDER_NO"].ToString().Trim() != "")
{
    // 只有订单号不为空，才会执行红冲
    if (THROW_FLAG == "Y" || THROW_FLAG == "Z" || THROW_FLAG == "T")
    {
        f_ymre_toERP(...);  // 红冲处理
    }
}
```

**排查SQL：**

```sql
-- 检查批次订单号
SELECT IN_BATCH_NO, BUY_ORDER_NO, THROW_FLAG, BASE_RESOURCE
FROM TYMREPCINFO
WHERE IN_BATCH_NO = '批次号';
```

#### 原因3：抛帐状态不符合条件

**问题描述：** `THROW_FLAG` 不在允许撤回的范围

**代码要求：** 第118行

```cpp
if (THROW_FLAG != "T" && THROW_FLAG != "Y" && THROW_FLAG != "Z")
    → 不允许撤回
```

**排查SQL：**

```sql
-- 检查批次抛帐状态
SELECT IN_BATCH_NO, THROW_FLAG, ZANGU_FLAG, PSCS_PC_STATUS
FROM TYMREPCINFO
WHERE IN_BATCH_NO = '批次号';
```

#### 原因4：原抛帐记录的交易类型不是01B

**问题描述：** 原抛帐时使用的 `TRADE_TYPE` 不是 `'01B'`，导致红冲查询找不到记录

**排查SQL：**

```sql
-- 检查原抛帐记录的交易类型
SELECT IN_BATCH_NO, TRADE_TYPE, BALANCE_WGT, ENABLE_FLAG, 
       DST_STOCK_CODE, REC_CREATE_TIME
FROM TYMREE1 
WHERE IN_BATCH_NO = '批次号'
  AND ENABLE_FLAG = '1'
ORDER BY REC_CREATE_TIME DESC;
```

#### 原因5：目的库区不匹配

**问题描述：** 批次下的车辆 `DST_STOCK_CODE` 与原抛帐记录不一致

**排查SQL：**

```sql
-- 检查批次下所有车辆的目的库区
SELECT IN_BATCH_NO, WORK_SEQ_NO, DST_STOCK_CODE, BALANCE_WGT
FROM TYMREC1
WHERE IN_BATCH_NO = '批次号'
  AND RECV_CONFM_FLAG = 'Y';

-- 检查原抛帐记录的目的库区
SELECT DISTINCT IN_BATCH_NO, DST_STOCK_CODE, TRADE_TYPE
FROM TYMREE1
WHERE IN_BATCH_NO = '批次号'
  AND ENABLE_FLAG = '1';
```

#### 原因6：`f_ymre_toERP` 函数内部异常

**问题描述：** 函数调用失败，但错误被忽略或未正确返回

**可能原因：**

- 物流模式配置缺失或错误
- 成本中心配置缺失
- 账套配置问题
- 品名代码映射缺失

**排查方法：**

- 查看服务日志（`EDLog` 输出）
- 检查是否有异常被捕获但未抛出

### 11.3 完整排查步骤

#### 步骤1：检查批次基本信息

```sql
SELECT 
    P.IN_BATCH_NO,
    P.BUY_ORDER_NO,
    P.THROW_FLAG,
    P.ZANGU_FLAG,
    P.PSCS_PC_STATUS,
    P.BASE_RESOURCE,
    P.ENTR_FLAG,
    COUNT(T.WORK_SEQ_NO) AS 车辆数,
    SUM(T.BALANCE_WGT) AS 批次合计量
FROM TYMREPCINFO P
LEFT JOIN TYMREC1 T ON T.IN_BATCH_NO = P.IN_BATCH_NO
WHERE P.IN_BATCH_NO = '批次号'
GROUP BY P.IN_BATCH_NO, P.BUY_ORDER_NO, P.THROW_FLAG, 
         P.ZANGU_FLAG, P.PSCS_PC_STATUS, P.BASE_RESOURCE, P.ENTR_FLAG;
```

#### 步骤2：检查原抛帐记录

```sql
SELECT 
    E.IN_BATCH_NO,
    E.TRADE_TYPE,
    E.BALANCE_WGT,
    E.H2O_NET,
    E.ENABLE_FLAG,
    E.HC_FLAG,
    E.DST_STOCK_CODE,
    E.REC_CREATE_TIME,
    E.ERP_SEQ_NO
FROM TYMREE1 E
WHERE E.IN_BATCH_NO = '批次号'
  AND E.BALANCE_WGT > 0
ORDER BY E.REC_CREATE_TIME DESC;
```

#### 步骤3：检查是否已生成红冲记录

```sql
SELECT 
    E.IN_BATCH_NO,
    E.HC_FLAG,
    E.BALANCE_WGT,
    E.TRADE_TYPE,
    E.ENABLE_FLAG,
    E.REC_CREATE_TIME
FROM TYMREE1 E
WHERE E.IN_BATCH_NO = '批次号'
  AND E.HC_FLAG = 'Y'  -- 红冲标志
ORDER BY E.REC_CREATE_TIME DESC;
```

#### 步骤4：对比原抛帐与批次下车辆的目的库区

```sql
-- 原抛帐记录的目的库区
SELECT DISTINCT DST_STOCK_CODE, COUNT(*) AS CNT
FROM TYMREE1
WHERE IN_BATCH_NO = '批次号'
  AND ENABLE_FLAG = '1'
  AND BALANCE_WGT > 0
GROUP BY DST_STOCK_CODE;

-- 批次下车辆的目的库区
SELECT DISTINCT DST_STOCK_CODE, COUNT(*) AS CNT
FROM TYMREC1
WHERE IN_BATCH_NO = '批次号'
  AND RECV_CONFM_FLAG = 'Y'
GROUP BY DST_STOCK_CODE;
```

### 11.4 解决方案建议

#### 方案1：修复原抛帐记录

如果原抛帐记录存在但不符合查询条件：

- 检查并修正 `TRADE_TYPE`、`ENABLE_FLAG`、`DST_STOCK_CODE` 等字段
- 确保 `BALANCE_WGT > 0`

#### 方案2：手动补录红冲记录

如果确认需要红冲但系统未生成：

1. 查找原抛帐记录的关键信息
2. 手动插入红冲记录到 `TYMREE1`（设置 `HC_FLAG='Y'`，`BALANCE_WGT` 为负值）
3. **注意：** 此操作需谨慎，建议联系系统管理员

#### 方案3：重新抛帐后再撤回

如果原抛帐记录缺失：

1. 检查批次是否可以重新抛帐
2. 重新抛帐生成正确的 `TYMREE1` 记录
3. 再次执行撤回操作

### 11.5 预防措施

1. **定期检查：** 对已抛帐的批次，定期检查 `TYMREE1` 记录是否完整
2. **数据校验：** 批次抛帐后，验证 `TYMREE1` 记录的关键字段
3. **监控告警：** 对撤回操作失败的情况设置监控告警
4. **日志分析：** 定期分析服务日志，发现潜在问题
